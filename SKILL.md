# 辩论技能

用于编排结构化的多角色辩论。它可以从当前会话中实际可用的角色技能中自动选择合适的角色，以隔离模式或内联模式运行，并生成完整的辩论记录以及中立综合总结。

## 输入

从用户的调用中解析以下内容：
- **Topic（主题）** —— 被辩论的问题（必填）
- **Personas（角色）** —— 每一方你需要创建四个人，分别为 一辩，二辩，三辩，四辩。总共有两方，观点对立。如果用户明确点名了角色，就使用这些角色。如果未指定，就从当前会话中**实际可用**的角色技能里自动选择。使用例如正方一辩，正方二辩的代号（持方 + 辩位），这些角色实际存在。
- **Support（持方）**——正方或反方（必填）。如果用户明确指出持方，则最后要针对持方出具 一辩稿子，二辩稿子，三辩自证稿子，四辩结辩稿子，三质二质询稿，四质一质询稿。每个稿子单独一个文件。
- **Rounds（轮数）** —— 默认为 3（Opening -> Rebuttal -> Closing，即开场 -> 反驳 -> 结辩）。用户可以覆盖。
- **Mode（模式）** —— `isolated`（默认）或 `inline`。参见下方 **Mode Selection（模式选择）**。
- **Synthesis（综合总结）** —— 默认为开启。即最后会有一个中立主持人的总结。用户可以说 “no synthesis” 来跳过。

如果对话上下文中已经有一个明显的主题（例如，刚刚结束了一轮关于某个问题的并行独白），则把它视为主题，不必再次询问。

资料可以从 https://debatetimer.cn/ 上面寻找，关于辩论的知识请确保在开始前已经在网络上了解到如何执行。

在开始前请确认每一个辩手的人设，你可以在网上寻找知名的辩手作为人设和名称。

## 角色选择

优先级顺序：
1. 用户明确指定的角色
2. 从已安装/可用的角色技能中自动选择
3. 如果找不到至少 2 个合理角色，只询问一次

### 自动选择工作流

当用户没有指定角色时：

1. 检查当前会话中可用的角色技能。**只能使用当前环境中真实安装或暴露出来的技能；不要假设 Musk/Naval/Jobs 一定存在。**
2. 根据主题推断 2 个所需的视角类别。优先选择差异明显的视角，而不是近似重复的视角。
3. 根据名称和描述，将可用的角色技能匹配到这些类别上。
4. 选择能产生真实分歧的最小强组合，通常是 4个角色。
5. 如果置信度较低，开始前简短说明你的假设；如果匹配严重失真，则询问一次。

### 视角类别

可使用如下轻量启发式规则：

- 创业、职业、杠杆、财富、自由、激励 -> 具有杠杆或职业框架的创始人 / 投资人 / 经营者
- 产品、UX、品味、品牌、简化、消费者体验 -> 产品 / 品味 / 设计型角色
- 工程、第一性原理、系统、制造、成本结构 -> 建造者 / 技术 / 第一性原理型角色
- 哲学、决策、人生策略、自我认知 -> 反思 / 哲学型角色
- 市场、战略、权力、竞争、执行 -> 战略型或竞争型角色

### 匹配规则

- 用户明确指定的角色永远优先。**绝不要覆盖用户点名的角色。**
- 优先选择那些描述中明显体现出**独特世界观**的角色技能，而不只是主题相近。
- 需要选择四个同一视角为一组，两组观点对立代表正反双方。
- 如果只有一个角色明显契合，不要硬凑辩论；应询问一次，或者说明辩论至少需要两个不同视角。
- 如果根本发现不了任何角色技能，简要说明后停止，**不要凭空捏造角色。**
- 如果有多个可行组合，选择**视角多样性最高**的一组并直接开始。
- 在 debaters/ 文件夹下有各类辩手，请你从 debater.md 中寻找合适的角色。
- 优先让辩手打推荐的辩位。

## 模式选择

**`isolated`（隔离模式，默认）** —— 每个角色都在自己的子代理中运行，使用 Agent 工具。编排器本身**不直接撰写角色回应**；它只负责组装辩论记录并生成最终综合总结。  
- **优点**：结构独立。每个角色只会看到 `角色技能 + 主题 + 之前的记录（作为数据）`。不会继承编排器对其他发言者的解释层。
- **缺点**：开销比内联更高；编排器无法在回复中途纠偏；长辩论中，角色可能会越来越锚定在自己早先的措辞上。
- **适用场景**：用户担心编排器偏见；角色之间词汇/世界观重叠较多，容易互相污染；或者你希望看到更尖锐的分歧。

**`inline`（内联模式）** —— 顺序调用角色技能，在主对话上下文中执行。编排器直接撰写每个角色的回应，但会使用刚刚加载的技能设定。  
- **优点**：快，节省 token，并允许实时质量控制（可以在中途发现角色漂移并重新锚定）。
- **缺点**：前面角色的论点会留在编排器上下文中，因此编排器对先前发言的理解，会微妙地影响后续角色的表达。存在框架偏差风险。
- **适用场景**：速度更重要，或用户希望进行迭代式质量控制。

默认使用 `isolated`。当用户说 “inline”、“fast mode”、“single-context” 或 “mode=inline” 时，切换为 `inline`。

## 执行协议

**顺序轮换**：每一轮都轮换发言顺序，避免在自由辩论中某个角色永远最后发言。对于 3 个角色 A/B/C：  
R1 = A->B->C，R2 = B->C->A，R3 = C->A->B。

### 内联模式

对每一轮、每一个按顺序发言的角色：

1. 通过 Skill 工具调用该角色技能（例如：`Skill(skill="elon-musk-perspective")`）。
2. 技能加载后，直接在主对话中撰写该角色的回应。你要满足的提示内容包括：
   - 辩题
   - 当前轮次名称和指令（开场 / 反驳 / 结辩 —— 见上文轮次结构）
   - 所有先前轮次的完整记录，并按角色标注
   - 技能中的角色扮演规则（必须严格遵守 —— 使用“我”，保持角色内，不要元评论）
3. 然后进入下一个角色。

**关键点**：每一轮都要**重新 fresh 调用**该角色技能。不要试图“凭记忆”沿用他们之前的声音——角色的扮演 DNA 存在于技能文件里，长上下文中会逐渐衰减。

### 隔离模式

在每一轮内并行运行子代理。同一轮中的每个角色都只看到**之前轮次**的共享记录；**同一轮其他角色的回应不能先被看到再作答**。这是隔离模式下默认的公平规则，因为它减少顺序偏差，也能加快执行，同时保留跨轮交锋。

**优先实现方式**：在辩论开始时，为每个角色各启动一个持久代理，然后在后续轮次中持续向这些代理发送后续消息。在像 Claude Code 这样暴露 agent ID 且支持“向现有代理发送消息”的平台上，应始终优先使用这一方式，而不是每一轮重新生成代理。

**Claude Code 说明**：在 Claude Code 中，向现有子代理继续发送消息依赖 `SendMessage`，而它当前要求启用 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`。如果该标志未开启，则退回到“每轮重新生成”的方式。

**回退方案**：如果平台无法继续使用现有代理，则允许每轮重新生成，但这是较差的备选方案，因为它会重复加载技能并在初始化上浪费 prompt token。

**启动阶段 / 第 1 轮 —— 并行持久代理**：并行为每个角色各启动一个代理。每个代理应加载其角色技能一次，将主题保留在记忆中，并在首次响应中返回自己的开场陈述。提示模板如下：

```text
你正在执行一场角色辩论。你的**唯一任务**是以 {persona} 的风格作答。

步骤 1：通过调用 Skill(skill="{persona_skill_name}") 加载该角色技能。严格遵守其角色扮演规则——使用“I/我”，保持角色内，不要写元评论，也不要写“如果是 {persona}，他会说”。

步骤 2：以 {persona} 的身份回应以下提示：

TOPIC: {topic}

ROUND: 开场陈述。陈述你对该主题的立场。一个核心论点加上理由。150–300 词。当前还没有先前论点可回应——这是你的开场。

返回**仅限**角色本人的回应。不要前言，不要总结，不要舞台说明。该回应将被原样插入辩论记录中。
```

**第 2 轮 —— 并行后续消息**：并行复用同一批持久代理。因为每个代理已经知道主题，也记得自己第 1 轮的回答，所以只需发送**其他角色**在第 1 轮中的回应，除非平台不支持持久代理续接。提示模板如下：

```text
你正在执行一场角色辩论。你的**唯一任务**是以 {persona} 的风格作答。

请继续以角色身份参与辩论。除非有必要，不要重新加载或总结你之前的回答。你已经知道：
- 主题
- 你自己在第 1 轮的立场
- 你的角色规则

以下是其他角色在第 1 轮中的发言：

### {Other Persona 1}
{Other Persona 1 R1 response}

### {Other Persona 2}
{Other Persona 2 R1 response}

ROUND 2 INSTRUCTIONS: 你现在要进行的是**反驳**。你必须：
- 直接点名并挑战上面至少一个其他角色的具体论点。
- 在被攻击处为你自己的第 1 轮立场辩护或加以修正。
- 不要重复第 1 轮——要真正回应别人说了什么。
- 150–300 词。

返回**仅限**角色本人的回应。不要前言，不要舞台说明。
```

**第 3 轮 —— 并行后续消息**：再次并行复用同一批持久代理。每个代理已经记得自己前两轮的发言，因此只需发送其他角色在第 2 轮中的回应；如果平台不支持持久代理续接，则不适用：

```text
请继续以角色身份参与辩论。你已经知道主题以及你自己此前的立场。

以下是其他角色在第 2 轮中的发言：

### {Other Persona 1}
{Other Persona 1 R2 response}

### {Other Persona 2}
{Other Persona 2 R2 response}

ROUND 3 INSTRUCTIONS: 这是你的**结辩**。你必须：
- 承认至少一个其他角色说对了的具体点（要求智识诚实——不能死扛）。
- 给出经过这次交锋后更清晰、更锋利的最终立场。
- 150–300 词。
```

**所有轮次完成之后**：由编排器（主 Claude）组装完整记录，并以内联方式撰写 **Synthesis（综合总结）** 部分（综合总结始终以内联方式完成，绝不使用子代理——因为它必须在统一视角下看到全部内容）。

**捕获子代理输出**：子代理会把其回应作为工具结果返回。提取其中角色本人的回应，并原样插入辩论记录。如果某个子代理尽管被明确要求，仍然返回了前言或元评论，则先对同一持久代理重试一次，明确要求“只返回角色内回应”。只有在无法恢复时才重新生成代理。

**执行说明**：在隔离模式中，为每个角色维持一个持久代理贯穿整场辩论，并通过后续消息（`send message`、`continue agent` 或平台等价 API）在各轮继续使用它。这是 Claude Code 以及所有支持代理续接的平台的默认路径。除非平台不支持续接、角色严重漂移、或代理已不可用，否则**不要**每一轮都重新生成新代理。

**持久代理的 token 节约原则**：
- 启动一次，多次发消息。
- 每个代理只加载一次角色技能，不要每轮都加载。
- 在后续轮次中，只发送代理**尚不知道**的内容：主要是其他角色最近一轮的回应，以及该轮的具体指令。
- 如果辩论超过 5 轮，或者代理开始被自己早先的措辞固化，可考虑受控地重新生成，并附上紧凑回顾。

## 综合总结规则

如果开启综合总结，则在结尾加入一个中立主持人部分，要求：
- 点出核心分歧（而不是只说“大家都有道理”）
- 点出在多轮交锋中真正出现的共识
- 识别出哪个角色更新了自己的看法，以及更新了什么
- 不宣布“赢家”，但**可以**给出条件性建议路径（例如：“如果你处于 X 情境，更应倾向 Y”）
- **不要捏造角色没有持有的立场**，但**可以**将他们的论点重新组织、重构为更结构化的拆解、实用启发式或可执行决策框架

综合总结的声音应是主 Claude 的声音，而不是任何角色的声音。

## 长度约束

每个角色每轮：目标为 150–350 词。超过这个范围，辩论质量通常会迅速下降——角色会开始重复自己。如果某个角色技能倾向于长篇独白，请在该轮提示中明确写明“控制在 300 词以内，只讲一个核心论点”。

完整的 3 轮、3 角色辩论，输出大约 ≈ 2500–3000 词。如果用户要求的规模明显会超出这个范围（例如 5 个角色 × 5 轮），开始前应先提醒用户。

## 退出角色

在最终综合总结之后，所有角色默认都已退出角色状态。如果用户继续追问，则以正常 Claude 身份回答，除非他们再次调用某个角色。

## 需要避免的失败模式

- **并行独白**：如果第 2 轮及之后，角色没有点名并真正回应彼此的论点，那么这场辩论就是失败的。应重新提示该角色，并明确写明“你必须回应 X 关于 Y 的主张”。
- **虚假共识**：角色过快趋同，通常说明编排器把他们的声音软化了。应信任技能文件——如果一个角色本来会不同意另一个角色，就让这种分歧保留下来。
- **主题漂移**：每一轮的提示都必须重申原始主题。否则角色会逐渐跑题。
- **角色泄漏**：如果某个角色开始听起来像普通泛化回答，说明角色锚定没有做好。应重新调用或重新提示后重试。
